## 1. 执行策略

### 1.1 核心原则

1. **逐章推进**：从 Cap.1 开始，每章完成后验证 + 提交
2. **先做短篇+中篇，长篇最后统一做**：长篇留到全量短/中篇完成后集中处理
3. **注水不如短小**：Cap.1-4 词汇少时不硬推中篇
4. **保留已有成果**：Cap.11-12 已完成的不动（但需用新算法重验证）

### 1.2 执行顺序

| 优先级 | 章节 | 做哪些 | 理由 |
|:---:|------|------|------|
| 1 | Cap.1-4 | 短篇翻新 + 补缺 | 初学者最需要，词汇少、最快 |
| 2 | Cap.5-9 | 短篇翻新 + 升中篇（5:3） | 中篇可行 |
| 3 | Cap.10-20 | 短篇翻新 + 升中篇（先不管长篇） | 高章量大 |
| 4 | Cap.21+ | 短篇翻新 + 升中篇（先不管长篇） | 长尾 |
| 5 | 全量长篇 | Cap.10+ 统一做长篇（5:3:2 中的 2） | 最后做，确保质量 |

- 长篇 700-1000 词，允许自然延伸到 1200

## 3. 每篇重写工作流

```
1. 读取原故事 YAML + 正文
2. 保留元数据（story_id, title_la, title_zh, theme, style, genre, character_type, narrative_mode）
3. 确定目标层级：
   - Cap.1-4 → 短篇（150-349）
   - Cap.5-9 → 短篇或中篇（350-699），按 5:3 比例
   - Cap.10+ → 短篇或中篇（先不做长篇）
4. 扩充方式（保留核心情节）：
   - 增加对话回合
   - 增加感官细节（视觉/听觉/触觉/嗅觉）
   - 增加内心独白或人物反应
   - 增加场景切换或前后呼应
   - 不改变原 theme / genre / character_type / narrative_mode
5. 算法验证（evaluate_v2.py，80th 百分位 + 专有名豁免）
   - v2_level ≤ target_chapter + 2 即通过
   - 失败则用安全词表替换高章词（最多 2 轮）
6. 更新 YAML：
   - length_tier: "短篇" / "中篇" / "长篇"
   - word_count: 实际词数
   - status: "rewritten"
   - rewritten_at: ISO8601 时间戳
7. 写入文件，文件名用新后缀
```

### 文件命名规则

| 后缀 | 含义 | 示例 |
|------|------|------|
| `_brevis_NNN` | 短篇 | `Cap1_Roma_brevis_001.md` |
| `_medius_NNN` | 中篇 | `Cap7_Puer_et_canis_medius_001.md` |
| `_longus_NNN` | 长篇 | `Cap13_Mercator_longus_001.md` |

### 补缺规则（目标每章 ≥ 10 篇）

当某章不足 10 篇时，需新写短篇：
- 复用该章已用主题、风格、题材，但换不同人物/场景
- 优先补简单主题（自然、日常生活）——故事最易写、词汇最省
- 新故事同样过算法验证

## 6. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Cap.1-4 词汇不足 | 150-200 词紧凑短篇好过注水中篇 |
| 算法判高 | 80th + 专名豁免 + 安全词替换（最多 2 轮） |
| 批量质量下降 | 每 10 篇人工抽检 1 篇 |
| 上下文丢失进度 | 每章完成后 git commit |
| 长篇最后做耗时长 | 预留独立阶段，做完短/中篇后专心投入 |
