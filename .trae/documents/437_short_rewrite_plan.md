## 3. 每篇重写工作流

```
1. 读取原故事 YAML + 正文
2. 保留原 metadata（story_id, title_la, title_zh, theme, style, genre, character_type, narrative_mode）
3. 确定目标层级（按上表分配）
4. 用目标章节的累计词汇表重写正文（Cap.N 的词汇 = Cap.1 至 Cap.N 的并集）
5. 算法验证（evaluate_v2.py）— v2_level ≤ target_chapter + 2 即通过
6. 更新 YAML（evaluated_chapter, v2_rate, v2_oov, word_count）
7. 更新 length_tier 为新的层级标签
8. 写入文件（覆盖原文件或生成新文件 → 决定：覆盖原文件，但文件名中的层级后缀改为新值）
```

### 文件命名规则更新

| 旧后缀 | 新后缀 |
|--------|--------|
| `_brevis_`（短篇） | `_medius_`（中篇） |
| - | `_longior_`（中长篇） |
| - | `_longus_`（长篇） |
| - | `_longissimus_`（超长篇） |

### 元数据变更

- `length_tier`: `"短篇"` → `"中篇"` / `"中长篇"` / `"长篇"` / `"超长篇"`
- `status`: 改为 `"rewritten"`
- 新增 `rewritten_from: "brevis"` 字段
- 新增 `rewritten_at: "<timestamp>"`

## 4. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 上下文压缩丢失进度 | 每次完成一章立即更新 `rewrite_progress.json`（独立追踪文件）|
| 低章节词汇不足撑长度 | 用修辞排比、对话展开、感官细节填充——不强行堆生词 |
| 算法判分过高 | OOV 词置换为同义近义已学词，最多 2 轮迭代 |
| 故事质量下降（注水） | 每条重写后人工抽检（每 10 篇抽 1 篇阅读全文）|
| Cap.10/13/16/19 数量大 | 分成子批次，每批次完成后 commit |

## 6. 执行指令（给 AI）

> 以下指令供执行阶段读取。规划阶段请勿执行。

1. 每次只处理**一个阶段的一个子批次**，完成后回报进度、提交 git
2. 重写脚本命名：`rewrite_<scope>.py`，放在 `AI_reader/` 下
3. 每篇重写的拉丁正文写在脚本的 `STORIES` 字典中（与 `fill_all_gaps.py` 同模式）
4. 重写原则：
   - **保留原故事核心情节与立意**，只扩充篇幅
   - 扩充方式：增加对话回合、增加感官细节描写、增加人物内心独白、增加场景切换
   - 不随意改变原 theme / genre / character_type / narrative_mode
5. 遇到无法用目标词汇表达的情节，换一种说法而非强行生造 OOV 词
6. 每次开始新批次前，先读取 `rewrite_progress.json` 确认哪些已完成、哪些待处理
