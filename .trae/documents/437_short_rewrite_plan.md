## 1. 执行策略

### 1.1 核心原则

1. **先翻新，再升级**：先把短篇改写为更丰富的短篇（150-349 词），然后在允许的章节升级为中篇/长篇
2. **注水不如短小**：Cap.1-4 词汇不足时不硬推中篇，一篇 180 词的紧凑短篇好过 350 词的重复排比
3. **逐章推进**：每章完成后算法验证 + 提交，不跨章并行
4. **保留已有成果**：Cap.11-12 已完成的中篇/长篇不动

### 1.2 优先级

| 优先级 | 章节 | 理由 |
|:---:|------|------|
| 1 | Cap.1-5 | 初学者最需要阅读量，词汇少、故事短、改起来最快 |
| 2 | Cap.6-10 | 词汇开始丰富，中篇可达 |
| 3 | Cap.13-15 | Cap.11-12 已完成，继续推进 |
| 4 | Cap.16+ | 高章目前全短篇，需大量工作 |

## 3. 每篇重写工作流

```
1. 读取原故事 YAML + 正文
2. 保留元数据（story_id, title_la, title_zh, theme, style, genre, character_type, narrative_mode）
3. 确定目标层级：
   - Cap.1-4 → 短篇（150-349）
   - Cap.5-9 → 短篇或中篇（350-699）
   - Cap.10+ → 短篇/中篇/长篇（150-1000+）
   每篇目标由故事自然长度决定，不做强制
4. 改写核心情节保留，扩充：
   - 增加对话回合
   - 增加感官细节（视觉/听觉/触觉）
   - 增加内心独白或人物反应
   - 增加场景切换或前后呼应
   - 不改变原 theme / genre / character_type / narrative_mode
5. 算法验证（evaluate_v2.py，80th 百分位 + 专有名豁免）
   - v2_level ≤ target_chapter + 2 即通过
   - 低 chapter 故事若 v2_level 超标，用安全词表替换高章词
6. 更新 YAML：
   - length_tier: "短篇" / "中篇" / "长篇"
   - status: "rewritten"
   - rewritten_from: "brevis"（如原本是短篇）
   - rewritten_at: ISO8601 时间戳
   - word_count: 实际词数
7. 写入文件，文件名用新后缀
```

### 文件命名规则

| 后缀 | 含义 | 示例 |
|------|------|------|
| `_brevis_NNN` | 短篇 | `Cap1_Roma_brevis_001.md` |
| `_medius_NNN` | 中篇 | `Cap7_Puer_et_canis_medius_001.md` |
| `_longus_NNN` | 长篇 | `Cap13_Mercator_longus_001.md` |

## 5. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Cap.1-4 词汇不足导致注水 | 150-200 词的紧凑短篇好过 350 词的重复排比 |
| 算法判高 | 80th 百分位 + 专有名豁免 + 安全词替换（最多 2 轮迭代）|
| 批量生成质量下降 | 每 10 篇人工抽检 1 篇 |
| 上下文压缩丢失进度 | 每章完成后立即 git commit |
